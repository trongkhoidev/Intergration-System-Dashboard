# API Integration Plan Document

## 1. Executive Summary
The Integration-445 project serves as a comprehensive "Integrated HR & Payroll Management Dashboard" designed to bridge the data gap between legacy HR management systems and modernized payroll processing engines. This document defines the strategic roadmap, architecture, and deployment plan for orchestrating data flows across a heterogeneous database environment utilizing Microsoft SQL Server and MySQL. 

The primary objective is to establish a secure, performant, and resilient API layer (built on Python Flask) that guarantees data consistency across distinct database systems while serving a unified React.js frontend.

## 2. Integration Architecture Strategy
The system adopts a **Middleware API Gateway Architecture**, where the Flask application acts as the central orchestration engine.
- **System of Record A (HR Core & Auth):** Microsoft SQL Server (`HUMAN` and `HUMAN_AUTH` databases).
- **System of Record B (Payroll Processing):** MySQL (`PAYROLL` database).
- **Client Application:** React.js Single Page Application (SPA).
- **Communication Protocol:** RESTful HTTP/HTTPS.
- **Data Format:** JSON (JavaScript Object Notation).

## 3. Phased Integration Roadmap

### Phase 1: Infrastructure and Database Initialization
- **Objective:** Provision the underlying data stores and establish connectivity.
- **Deliverables:**
  - Deployment of SQL Server schemas (`Departments`, `Positions`, `Employees`, `Dividends`, `SystemUsers`, `AuditLogs`).
  - Deployment of MySQL schemas (`employees_payroll`, `attendance`, `salaries`).
  - Configuration of environment variables (`.env`) for cross-database access control.
  - Establishment of connection pools using `pyodbc` and `mysql.connector`.

### Phase 2: Core API & Synchronization Engine
- **Objective:** Develop the RESTful endpoints and the dual-write synchronization logic.
- **Deliverables:**
  - **Auth Layer:** Implementation of JWT-based authentication, PBKDF2 password hashing, and the OTP password recovery pipeline.
  - **HR Synchronization:** Implementation of a "Pseudo 2-Phase Commit" for Employee creation/updating. Operations commit to SQL Server first, followed by a synchronous propagation to MySQL `employees_payroll`.
  - **Aggregation Services:** Endpoints (`/api/dashboard/stats`, `/api/alerts`) that query both SQL Server and MySQL simultaneously, aggregate the data in-memory within Python, and return unified JSON payloads.

### Phase 3: Frontend Consumption and RBAC Enforcement
- **Objective:** Integrate the React client with the backend APIs under strict Role-Based Access Control (RBAC).
- **Deliverables:**
  - Development of Axios HTTP interceptors to append JWT Bearer tokens to all outbound requests and handle `401 Unauthorized` token expirations.
  - Implementation of dynamic sidebar rendering based on JWT payload claims (`Admin`, `HR`, `Payroll`, `Employee`).
  - Binding API responses to UI components (Data Grids, Recharts, KPI Cards).

### Phase 4: Quality Assurance and System Hardening
- **Objective:** Ensure system resilience against data inconsistency and security threats.
- **Deliverables:**
  - Integration testing of the database synchronization mechanisms (simulating network failures during MySQL syncs).
  - Validation of the brute-force protection mechanism (15-minute account lockouts).
  - Comprehensive Audit Log testing to ensure all critical state-changing actions are recorded.

## 4. Technical Prerequisites & Dependencies
- **Backend:** Python 3.9+, Flask, PyJWT, Werkzeug (Security), PyODBC, MySQL-Connector-Python.
- **Frontend:** Node.js, React 18+, Vite, Axios, TailwindCSS (or similar UI framework).
- **Database Drivers:** ODBC Driver 17/18 for SQL Server.

## 5. Risk Management & Mitigation Strategies

| Risk Identifier | Risk Description | Probability | Impact | Mitigation Strategy |
| :--- | :--- | :---: | :---: | :--- |
| **RSK-01** | **Cross-Database Transaction Failure:** Network interruption after SQL Server commit but before MySQL commit. | Medium | High | The backend employs a fallback logging mechanism. If the MySQL sync fails, a `[WARN]` is logged, allowing for eventual consistency correction. Future versions will implement a message queue (e.g., RabbitMQ) for guaranteed delivery. |
| **RSK-02** | **Unauthorized Privilege Escalation:** Users manipulating APIs to access unauthorized modules. | Low | Critical | Strict `@require_auth` decorators on all endpoints. Server-side validation of JWT claims overrides any client-side state manipulation. |
| **RSK-03** | **Database Connection Saturation:** High concurrent requests exhausting database connection limits. | Medium | Medium | Implement connection pooling and set explicit `connect_timeout` (5s) for database drivers to fail fast and free up threads. |
| **RSK-04** | **Data Leakage via APIs:** APIs returning excessive data payloads to the client. | Low | High | API responses are explicitly mapped to return only necessary fields (e.g., `PasswordHash` is never serialized into JSON responses). |
