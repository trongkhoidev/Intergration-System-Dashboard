# Database Design & Data Mapping

## 1. System Topology Overview
The architecture is inherently decentralized at the data layer to enforce separation of concerns between core human resources administration, system security, and high-frequency financial processing.
- **Node A (SQL Server):** Hosts the `HUMAN` database for core personnel records and the `HUMAN_AUTH` database for RBAC and security auditing.
- **Node B (MySQL):** Hosts the `PAYROLL` database dedicated to attendance tracking and salary generation.

## 2. Database Physical Schemas

### 2.1 SQL Server - Database: `HUMAN`
This schema acts as the ultimate source of truth for employee identity.
* **Departments**: `DepartmentID` (INT, PK, Identity), `DepartmentName` (NVARCHAR 100, Not Null).
* **Positions**: `PositionID` (INT, PK, Identity), `PositionName` (NVARCHAR 100, Not Null).
* **Employees**:
  * `EmployeeID` (INT, PK, Identity)
  * `FullName` (NVARCHAR 100, Not Null)
  * `Email` (NVARCHAR 100, Unique)
  * `DateOfBirth` (DATE)
  * `Gender` (NVARCHAR 20)
  * `PhoneNumber` (NVARCHAR 20)
  * `HireDate` (DATE)
  * `DepartmentID` (INT, FK -> Departments)
  * `PositionID` (INT, FK -> Positions)
  * `Status` (NVARCHAR 50, Default 'Active') - *Crucial for Soft Deletion workflows.*
* **Dividends**: `DividendID` (INT, PK, Identity), `EmployeeID` (INT, FK -> Employees), `Amount` (DECIMAL 18,2).

### 2.2 SQL Server - Database: `HUMAN_AUTH`
Isolated to ensure that a breach in the HR system does not immediately compromise authentication tokens.
* **SystemUsers**:
  * `UserID` (INT, PK, Identity)
  * `Username` (NVARCHAR 50, Unique)
  * `Email` (NVARCHAR 100)
  * `PasswordHash` (NVARCHAR 255) - *PBKDF2 HMAC SHA256.*
  * `Role` (NVARCHAR 20, Default 'User') - *Values: Admin, HR, Payroll, Employee.*
  * `FailedAttempts` (INT)
  * `LockedUntil` (DATETIME)
  * `EmployeeID` (INT, Nullable) - *Links the user account to the physical Employee record.*
  * `OTPCode` (NVARCHAR 6), `OTPExpiry` (DATETIME), `OTPAttempts` (INT)
  * `ResetToken` (NVARCHAR 255), `ResetTokenExpiry` (DATETIME)
* **AuditLogs**: `LogID` (INT, PK), `Username` (NVARCHAR 50), `Action` (NVARCHAR 100), `Details` (NVARCHAR MAX), `Timestamp` (DATETIME).

### 2.3 MySQL - Database: `PAYROLL`
Optimized for financial aggregations and time-series attendance tracking.
* **employees_payroll**:
  * `EmployeeID` (INT, PK) - *Non-auto-increment. Directly mapped from SQL Server.*
  * `FullName` (VARCHAR 100)
  * `DepartmentID` (INT)
  * `PositionID` (INT)
  * `Status` (VARCHAR 50, Default 'Active')
* **attendance**: `AttendanceID` (INT, PK, Auto-Increment), `EmployeeID` (INT, FK), `AttendanceMonth` (DATE), `WorkDays` (INT), `LeaveDays` (INT), `AbsentDays` (INT).
* **salaries**: `SalaryID` (INT, PK, Auto-Increment), `EmployeeID` (INT, FK), `SalaryMonth` (DATE), `BaseSalary` (DECIMAL 18,2), `Bonus` (DECIMAL 18,2), `Deductions` (DECIMAL 18,2), `NetSalary` (DECIMAL 18,2).

## 3. Data Mapping & Synchronization Logic

### 3.1 The `employees_payroll` Mirror Table
Because MySQL's `salaries` and `attendance` tables require an `EmployeeID` foreign key to generate reports, but joining across distinct RDBMS engines (SQL Server <-> MySQL) is highly inefficient, the `employees_payroll` table serves as a localized, read-optimized mirror of the SQL Server `Employees` table.

### 3.2 Transactional Synchronization Workflow
When a new employee is registered via `POST /api/employees`:
1. **Begin Transaction:** Python establishes connection to `HUMAN` with `autocommit=False`.
2. **Execute Insert A:** Record is inserted into `HUMAN.Employees`.
3. **Capture Key:** The newly generated `IDENTITY` value (`EmployeeID`) is captured via `OUTPUT INSERTED.EmployeeID`.
4. **Commit A:** The transaction on SQL Server is committed.
5. **Execute Insert B:** Python establishes connection to MySQL `PAYROLL`.
6. **Propagate:** An insert is executed on `PAYROLL.employees_payroll` forcing the `EmployeeID` PK to match the exact value captured from SQL Server.
7. **Commit B:** MySQL transaction is committed.

### 3.3 Conflict Resolution & Data Constraints
- **Soft Deletion Constraint (BR-05):** If an employee has received financial compensation (represented by records in the `HUMAN.Dividends` table), their profile **cannot be permanently deleted**. The API forces a `DELETE` request to execute an `UPDATE Employees SET Status = 'Inactive'` instead, which is then synchronized to MySQL.
- **Isolation of Payroll Logic:** Updates to an employee's profile (`PUT /api/employees`) sync their Name and Department to MySQL, ensuring payroll reports display accurate, up-to-date names without needing to query SQL Server.
