# System Test Plan Document

## 1. Introduction
This comprehensive Test Plan details the quality assurance strategy for the **Integration-445** system. The objective is to rigorously validate the stability, security, and data integrity of the integration layer connecting the Microsoft SQL Server (HR Core) and MySQL (Payroll) databases through the Python Flask API.

## 2. Testing Scope & Objectives

### 2.1 In-Scope Components
1. **Authentication & Authorization Security:** Login flows, JWT issuance/validation, Brute-force account lockouts, and strict Role-Based Access Control (RBAC).
2. **Cross-Database Data Synchronization:** Real-time dual-writing processes between `HUMAN.Employees` (SQL Server) and `PAYROLL.employees_payroll` (MySQL).
3. **Data Aggregation Services:** APIs that query multiple database engines simultaneously to formulate unified dashboard statistics and alerts.
4. **Soft Deletion Constraints:** Business logic preventing the hard deletion of human resources records that possess linked financial/dividend data.
5. **OTP Recovery Pipeline:** The end-to-end forgot password flow including expiration timing and attempt limiting.

### 2.2 Out-of-Scope Components
1. Stress testing beyond 500 concurrent connections.
2. Penetration testing of the physical underlying database servers.
3. Third-party SMTP server uptime and deliverability (Mock email testing is in-scope).

## 3. Test Strategy & Levels

### 3.1 Unit Testing (Level 1)
- **Target:** Core Python helper modules (`jwt_utils.py`, `config.py`).
- **Objective:** Validate cryptographic hashing functions, token generation algorithms, and database connection string parsing.

### 3.2 API Integration Testing (Level 2)
- **Target:** RESTful Endpoints (`router.py`).
- **Tooling:** Postman / Newman.
- **Objective:** Verify HTTP status codes, JSON schema structures, and database state mutations. Specifically, deliberately causing a failure in the MySQL connection during an Employee creation to ensure the system handles the exception gracefully without crashing the SQL Server transaction.

### 3.3 Security & Penetration Testing (Level 3)
- **Target:** RBAC Middleware and SQL Queries.
- **Objective:** 
  - Send requests to restricted endpoints (e.g., `/api/users`) using a token mapped to the `Employee` role. Expect a strict `403 Forbidden`.
  - Attempt SQL Injection by sending payload strings (e.g., `' OR 1=1 --`) via the login or employee search APIs. Expect `401 Unauthorized` or sanitized query execution.

### 3.4 System/End-to-End (E2E) Testing (Level 4)
- **Target:** React Frontend -> Flask API -> Database workflow.
- **Objective:** Validate the user journey from logging in, navigating the sidebar (verifying UI component restriction based on role), modifying an employee, checking the audit logs, and logging out securely.

## 4. Defect Management & Risk Assessment

### 4.1 Defect Severity Classification
- **Critical:** System crash, Data corruption between SQL and MySQL, Security bypass. (Resolution SLA: Immediate).
- **High:** Major API endpoint failure (`500 Error`), False positive lockouts. (Resolution SLA: 24 Hours).
- **Medium:** UI misalignments, incorrect dashboard calculations. (Resolution SLA: Next sprint).

### 4.2 Key Risks & Mitigations
| Risk Factor | Impact | Mitigation Strategy |
| :--- | :---: | :--- |
| **Token Hijacking** | Critical | Enforce short-lived JWTs. Future iteration to implement token blacklisting in `HUMAN_AUTH`. |
| **Partial Sync Failure** | High | Implement robust `try/except` handling during cross-DB writes. Log failures explicitly via `log_audit()` for manual intervention. |
| **Test Data Contamination** | Medium | Utilize distinct testing databases (`HUMAN_TEST`, `PAYROLL_TEST`) completely isolated from production datasets. |

## 5. Entry & Exit Criteria
- **Entry Criteria:** Backend APIs deployed locally, test databases populated with mock data (at least 1 Admin user, 10 Employees, 50 Salary records).
- **Exit Criteria:** 100% execution of Level 2 & 3 test cases. Zero 'Critical' or 'High' defects remaining open. RBAC matrix fully verified.
