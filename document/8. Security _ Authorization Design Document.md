# Security & Authorization Design Document

## 1. Introduction
This document outlines the holistic security framework for the Integration-445 system. It covers data protection at rest and in transit, authentication flow integrity, Role-Based Access Control (RBAC) enforcement, and compliance-driven audit mechanisms.

## 2. Threat Modeling & Mitigation
The system is designed to withstand standard web vulnerabilities as outlined by OWASP:
1. **SQL Injection (SQLi):** Mitigated across both SQL Server and MySQL by exclusively using parameterized queries (`?` in PyODBC, `%s` in MySQL) within `router.py`. String concatenation is strictly prohibited in SQL execution blocks.
2. **Brute Force & Credential Stuffing:** Mitigated via an aggressive Account Lockout Policy managed within the `SystemUsers` table.
3. **Session Hijacking / CSRF:** Mitigated via stateless JWTs. Tokens are signed with a strong cryptographic secret and have defined lifespans.
4. **Data Tampering:** Financial records and audit logs are restricted at the DB level and protected by strict RBAC at the API layer.

## 3. Authentication Subsystem
The authentication layer resides entirely within the isolated `HUMAN_AUTH` database.

### 3.1 Password Cryptography
Passwords are never stored in plain text. The system utilizes the `werkzeug.security` library.
- **Algorithm:** PBKDF2 (Password-Based Key Derivation Function 2).
- **Hash Function:** SHA-256.
- **Salting:** A unique salt is automatically generated and appended to every password before hashing, rendering rainbow-table attacks mathematically infeasible.

### 3.2 Account Lockout Mechanism (Brute-Force Protection)
1. Upon `POST /api/login`, the system queries `SystemUsers`.
2. The `LockedUntil` timestamp is verified. If `LockedUntil > CurrentTime`, an HTTP 423 is returned instantly.
3. If the password hash verification fails, `FailedAttempts` is incremented.
4. Upon reaching **3 consecutive failed attempts**, `LockedUntil` is set to `CurrentTime + 15 minutes`.
5. A successful login resets `FailedAttempts` to 0.

## 4. Authorization & RBAC
Access control is enforced via the `@require_auth(allowed_roles=[...])` Python decorator.
- The JWT is parsed, and the `role` claim is extracted.
- If the `role` is not within the `allowed_roles` array, the API immediately halts execution and returns HTTP 403 Forbidden.

**RBAC Matrix:**
| Feature / Module | Admin | HR | Payroll | Employee |
| :--- | :---: | :---: | :---: | :---: |
| **Login / Logout** | Yes | Yes | Yes | Yes |
| **View Dashboard Stats**| Yes | Yes | Yes | No |
| **CRUD Employees** | Yes | Yes | No | No |
| **Update Payroll** | Yes | No | Yes | No |
| **View All Reports** | Yes | Yes (HR only)| Yes (PR only)| No |
| **Manage Users** | Yes | No | No | No |
| **Self-Service Profile**| Yes | Yes | Yes | Yes |

## 5. Secure Password Recovery (OTP Flow)
To handle forgotten passwords securely without administrative intervention:
1. User requests recovery. System generates a **cryptographically secure 6-digit random number**.
2. OTP is stored in `SystemUsers` with a strict **10-minute expiry (`OTPExpiry`)**.
3. User submits OTP. The system tracks `OTPAttempts`. Over 5 failed attempts completely invalidates the OTP block to prevent brute-forcing the 6-digit space.
4. Validated OTP yields a temporary, one-time-use UUID `ResetToken` valid for 15 minutes to perform the actual password update.

## 6. Audit & Accountability (FR15)
To satisfy compliance and traceability requirements, all sensitive state mutations invoke the `log_audit(action, username, details)` helper.
- Writes to `HUMAN_AUTH.AuditLogs`.
- Captures the exact timestamp, the actor (`Username`), the operation (`EMPLOYEE_DEACTIVATED`, `PASSWORD_CHANGED`), and technical details.
- This creates an immutable trail of "Who did what, and when."
