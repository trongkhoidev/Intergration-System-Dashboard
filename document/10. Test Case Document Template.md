# System Test Case Document

## Overview
This document specifies the rigorous testing scenarios executed to validate the Integration-445 system. It is structured to provide clear instructions and definitive pass/fail criteria for both functional and security requirements.

---

### TC-SEC-01: Brute Force Account Lockout (BR-21)
* **Objective:** Validate that the system protects against brute-force attacks by locking accounts after excessive failed attempts.
* **Pre-conditions:** A valid user account `hr_admin` exists in `SystemUsers` with `FailedAttempts = 0`.
* **Test Steps:**
  1. Send `POST /api/login` with `{"username": "hr_admin", "password": "WrongPassword1"}`.
  2. Send `POST /api/login` with `{"username": "hr_admin", "password": "WrongPassword2"}`.
  3. Send `POST /api/login` with `{"username": "hr_admin", "password": "WrongPassword3"}`.
  4. Send `POST /api/login` with the **correct** password: `{"username": "hr_admin", "password": "CorrectPassword"}`.
* **Expected Result:** 
  - Attempts 1-2 return HTTP `401 Unauthorized`.
  - Attempt 3 returns HTTP `401 Unauthorized` with a message regarding account locking.
  - Attempt 4 returns HTTP `423 Locked` and denies access despite the correct password.
* **Status:** [ ] Pass / [ ] Fail

### TC-INT-01: Cross-Database Employee Provisioning
* **Objective:** Ensure data synchronization between the HR Database (SQL Server) and Payroll Database (MySQL).
* **Pre-conditions:** Authenticated as `Admin`. Databases are online.
* **Test Steps:**
  1. Send `POST /api/employees` with a valid JSON payload containing new employee details (`FullName: "Jane Doe"`, `Email: "jane@test.com"`).
  2. Query `SELECT * FROM HUMAN.Employees WHERE Email = 'jane@test.com'` in SQL Server.
  3. Note the newly generated `EmployeeID` from SQL Server.
  4. Query `SELECT * FROM PAYROLL.employees_payroll WHERE EmployeeID = <captured_id>` in MySQL.
* **Expected Result:** 
  - API returns HTTP `200 OK`.
  - Step 2 returns exactly 1 row.
  - Step 4 returns exactly 1 row with the identical `FullName` and `EmployeeID`.
* **Status:** [ ] Pass / [ ] Fail

### TC-BUS-01: Soft-Delete Financial Constraint (BR-05)
* **Objective:** Prevent the hard deletion or deactivation of an employee who has historical financial records.
* **Pre-conditions:** Authenticated as `HR`. Employee `E-100` exists and has 1 record in `HUMAN.Dividends`.
* **Test Steps:**
  1. Send `DELETE /api/employees/100`.
* **Expected Result:** 
  - API returns HTTP `409 Conflict`.
  - Response JSON includes message: `"Cannot deactivate: Employee has dividend records"`.
  - Employee `Status` remains `Active` in the database.
* **Status:** [ ] Pass / [ ] Fail

### TC-SEC-02: Strict Role-Based Access Control (RBAC) Validation
* **Objective:** Ensure lower-privileged users cannot access or manipulate administrative endpoints.
* **Pre-conditions:** Authenticated as `Payroll_User`. Token contains `role: "Payroll"`.
* **Test Steps:**
  1. Extract JWT token.
  2. Send `POST /api/users` (Endpoint to create new system accounts) attaching the token in the `Authorization: Bearer <token>` header.
* **Expected Result:** 
  - API immediately rejects the request with HTTP `403 Forbidden`.
  - No new user is created in the database.
* **Status:** [ ] Pass / [ ] Fail

### TC-AUTH-01: Secure OTP Generation and Attempt Limiting
* **Objective:** Verify the integrity of the password recovery workflow.
* **Pre-conditions:** User `test_user@company.com` exists.
* **Test Steps:**
  1. Send `POST /api/forgot-password` with `{"email": "test_user@company.com"}`.
  2. Query `SystemUsers` to extract the generated `OTPCode`.
  3. Send `POST /api/verify-otp` with an incorrect OTP 5 times.
  4. Send `POST /api/verify-otp` with the correct OTP (extracted in step 2) on the 6th attempt.
* **Expected Result:** 
  - Attempts 1-5 return `400 Bad Request` with remaining attempt count.
  - Attempt 6 returns `429 Too Many Requests` or `400 Bad Request` stating the OTP has been blocked/nullified due to excessive failures.
* **Status:** [ ] Pass / [ ] Fail

### TC-AUD-01: Immutable Audit Trail Generation
* **Objective:** Ensure critical actions are logged with accurate context.
* **Pre-conditions:** Authenticated as `Admin`.
* **Test Steps:**
  1. Send `PUT /api/employees/1` to change FullName to "Updated Name".
  2. Send `GET /api/audit-logs`.
* **Expected Result:** 
  - The API returns the list of logs.
  - The most recent log entry has `Username: "Admin"`, `Action` indicating an update or profile change, and a valid `Timestamp` within the last minute.
* **Status:** [ ] Pass / [ ] Fail
