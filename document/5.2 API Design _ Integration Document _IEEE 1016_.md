# API Design & Integration Document (IEEE 1016)

## 1. Introduction
This Software Design Description (SDD) aligns with the IEEE 1016 standard to document the architectural design of the API layer for the Integration-445 system. It defines the interfaces, protocols, and data payloads required to facilitate communication between the React SPA and the Dual-Database architecture.

## 2. API Architectural Design
The API follows RESTful principles:
- **Statelessness:** Every request from the client must contain all information necessary (JWT Token) to understand and process the request. No session state is held on the server.
- **Base URI:** `https://<server_domain>/api/`
- **Content Negotiation:** All requests and responses utilize `application/json`.
- **Standardized Responses:** Responses wrap data in a predictable structure, utilizing standard HTTP Status Codes.

### 2.1 HTTP Status Code Definitions
- `200 OK`: Request succeeded.
- `400 Bad Request`: Malformed JSON or missing required fields.
- `401 Unauthorized`: Missing, invalid, or expired JWT Token; invalid credentials.
- `403 Forbidden`: Valid token, but insufficient RBAC privileges for the requested resource.
- `404 Not Found`: The requested resource (e.g., Employee ID) does not exist.
- `409 Conflict`: Business logic violation (e.g., Email already exists, Soft-delete blocked by financial records).
- `423 Locked`: Account is locked due to excessive failed authentication attempts.
- `429 Too Many Requests`: Exceeded OTP entry limits.
- `500 Internal Server Error`: Unhandled exception or database connection failure.

## 3. Detailed Endpoint Specifications

### 3.1 Authentication & Security (`router.py`)

#### 3.1.1 POST `/api/login`
- **Purpose:** Authenticate users and issue JWT access tokens.
- **Roles Allowed:** Public
- **Request Payload:**
  ```json
  {
    "email": "user@company.com", 
    "password": "SecurePassword123"
  }
  ```
  *(Note: Supports both email and username formats).*
- **Response (200 OK):**
  ```json
  {
    "status": "success",
    "token": "eyJhbGciOiJIUzI1NiIsInR5c...",
    "user": {
      "user_id": 1,
      "username": "admin",
      "email": "admin@company.com",
      "role": "Admin",
      "employee_id": null,
      "full_name": "System Administrator"
    }
  }
  ```
- **Error Responses:** `401 Unauthorized` (Invalid credentials), `423 Locked` (Account locked).

#### 3.1.2 POST `/api/forgot-password` & `/api/verify-otp` & `/api/reset-password`
- **Purpose:** Secure multi-step password recovery flow using a 6-digit OTP.
- **Workflow:**
  1. `POST /api/forgot-password`: Expects `{"email": "..."}`. Returns `200 OK` (OTP sent via SMTP).
  2. `POST /api/verify-otp`: Expects `{"email": "...", "otp": "123456"}`. Returns `{"reset_token": "<UUID>"}`. Fails with `429` after 5 attempts.
  3. `POST /api/reset-password`: Expects `{"reset_token": "...", "new_password": "..."}`. Updates `PasswordHash`.

### 3.2 Human Resources Core (SQL Server Master)

#### 3.2.1 GET `/api/employees`
- **Purpose:** Retrieve full list of employees.
- **Roles Allowed:** Admin, HR, Payroll
- **Response Payload:** Array of Employee Objects (ID, FullName, Email, DOB, Gender, Phone, HireDate, Dept, Position, Status).

#### 3.2.2 POST `/api/employees`
- **Purpose:** Provision a new employee across both SQL Server and MySQL.
- **Roles Allowed:** Admin, HR
- **Request Payload:**
  ```json
  {
    "FullName": "John Doe",
    "Email": "john.doe@company.com",
    "DateOfBirth": "1990-01-01",
    "Gender": "Male",
    "PhoneNumber": "0123456789",
    "HireDate": "2024-01-01",
    "DepartmentID": 1,
    "PositionID": 2,
    "Status": "Active"
  }
  ```
- **Behavior:** Commits to `HUMAN.Employees` -> retrieves `EmployeeID` -> Commits to `PAYROLL.employees_payroll`.
- **Error Responses:** `409 Conflict` if Email is duplicate.

#### 3.2.3 DELETE `/api/employees/{id}`
- **Purpose:** Soft-delete an employee.
- **Roles Allowed:** Admin, HR
- **Behavior:** Checks `HUMAN.Dividends`. If records exist, returns `409 Conflict`. Otherwise, updates `Status = 'Inactive'` in both DBs.

### 3.3 Payroll & Attendance (MySQL Master)

#### 3.3.1 GET `/api/payroll`
- **Purpose:** Retrieve payroll data, joinable by department or month.
- **Roles Allowed:** Admin, HR, Payroll
- **Query Parameters:** `?month=May 2024&dept=IT`
- **Response Payload:**
  ```json
  [
    {
      "SalaryID": 1,
      "MonthYear": "May 2024",
      "FullName": "John Doe",
      "DepartmentName": "IT",
      "BaseSalary": 5000.00,
      "Bonus": 500.00,
      "Deductions": 100.00,
      "TotalSalary": 5400.00
    }
  ]
  ```

#### 3.3.2 PUT `/api/payroll/{id}`
- **Purpose:** Update specific salary parameters.
- **Roles Allowed:** Admin, HR, Payroll
- **Request Payload:** `{"BaseSalary": 5000, "Bonus": 500, "Deductions": 100}`
- **Behavior:** Backend automatically calculates `NetSalary`.

### 3.4 Aggregation & Dashboard Services

#### 3.4.1 GET `/api/alerts`
- **Purpose:** Real-time business intelligence alerts.
- **Behavior:** Issues parallel queries:
  - Queries SQL Server for Employee Birthdays matching current month.
  - Queries SQL Server for 1, 3, 5 year Work Anniversaries.
  - Queries MySQL `attendance` for Excessive Leave Days (>3 days).
  - Queries MySQL `salaries` for Salary Anomalies (>20% month-over-month variance).
- **Response Payload:** Array of Alert objects categorized by `severity` (info, warning, critical).

## 4. Security Considerations
- **SQL Injection:** All queries use parameterized inputs (e.g., `cur.execute("... WHERE Email = ?", (email,))` in PyODBC and `(%s)` in MySQL).
- **Token Storage:** Tokens should be handled securely by the client to mitigate XSS attacks.
- **Audit Logging:** Every state-altering API (POST, PUT, DELETE) and Auth API must invoke `log_audit()` to record the action in the `SystemUsers` schema.
